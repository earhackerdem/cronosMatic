## Context

The backend is a FastAPI application based on the Tiangolo full-stack template. Authentication today is exclusively email/password: the client POSTs credentials to `/api/v1/login/access-token` and receives a signed JWT. That JWT is then presented as a Bearer token on every subsequent request.

The `User` DB model stores a `hashed_password: str` field (non-nullable). All CRUD and security helpers assume this field exists. The frontend (React/Vite) manages the JWT in memory/localStorage and sends it as an Authorization header.

We need to add Google OAuth 2.0 as an alternative login path while keeping the email/password flow fully intact.

## Goals / Non-Goals

**Goals:**
- Two new endpoints: `GET /api/v1/auth/google/login` and `GET /api/v1/auth/google/callback`.
- Secure code exchange using Google's token endpoint, with CSRF protection via `state` parameter.
- Account provisioning: create a new `User` on first Google login; look up existing user on subsequent logins.
- Return the same native JWT the rest of the app already uses.
- Full unit-test coverage for both endpoints with Google API calls mocked.

**Non-Goals:**
- Linking a Google account to an existing email/password account (account merging).
- Support for other OAuth providers (GitHub, Apple, etc.) — this design does not preclude it but does not address it.
- Frontend implementation beyond what is needed for the backend redirect target.
- Refresh tokens or offline access scopes.

## Decisions

### 1. Use `httpx` directly instead of `authlib` or `python-social-auth`

`httpx` is already a dependency in the Tiangolo template (used by the test suite). The Google OAuth token exchange and user-info call are two straightforward HTTP requests; adding `authlib` for this would introduce a heavy dependency with its own abstraction layer for minimal benefit.

**Alternatives considered:**
- `authlib`: Full OAuth library with session management and provider configs. Overkill for a single provider; would require adapting its async patterns to FastAPI's sync route style.
- `python-social-auth`: Django-first, awkward in FastAPI contexts without significant adapter work.

### 2. Make `hashed_password` nullable, not generate a random hash

Setting `hashed_password = None` for Google-provisioned users gives us a reliable programmatic signal that an account has no local password. Generating a random hash works but silently allows the account to be "used" by password-reset flows — an unintended side effect.

**Migration impact:** One Alembic migration changes the column from `NOT NULL` to `NULLABLE`. Existing rows are unaffected (they all have hashed passwords). The `security.verify_password` call path already checks the user exists before calling `verify_password`, so no additional guard is needed beyond ensuring password-login is rejected for `hashed_password IS NULL` accounts.

**Alternatives considered:**
- Generate a random cryptographic hash: simpler schema change (none needed), but loses the ability to distinguish OAuth-only accounts cleanly.

### 3. Return JWT via redirect to `FRONTEND_HOST` with token as a query parameter

After successful callback, the server issues a redirect to `{FRONTEND_HOST}/auth/callback?access_token=<jwt>`. The frontend extracts the token from the URL and stores it normally.

**Alternatives considered:**
- Secure `HttpOnly` cookie: more secure (token not exposed to JS), but requires CORS `credentials: include`, cookie domain alignment between API and frontend domains, and frontend refactoring of the auth header flow. Higher complexity for v1.
- Return JSON in the callback response: works for same-domain setups but breaks the browser-redirect flow when the authorization server directly calls the callback URL.

### 4. CSRF protection via `state` parameter (server-generated, signed)

The `state` parameter is a URL-safe token generated by `secrets.token_urlsafe(32)` and stored in a short-lived signed cookie. The callback verifies the cookie matches the `state` query parameter before proceeding.

**Alternatives considered:**
- Store state in DB or cache (Redis): unnecessary operational dependency for a stateless check.
- Skip state validation: insecure — leaves the callback open to CSRF attacks.

### 5. New router file, not extending `login.py`

Google OAuth endpoints are semantically distinct from credential-based login. A separate `google_auth.py` router keeps `login.py` focused and makes the feature easy to find, extend, or remove.

## Risks / Trade-offs

- **Nullable password breaks password-login for OAuth users** → Mitigation: add an explicit guard in `login/access-token` that returns `400 Use Google login` when `hashed_password is None`.
- **State cookie not cleared after use** → Mitigation: delete the state cookie in the callback handler immediately after validation.
- **Google may return an unverified email** → Mitigation: check `email_verified: true` in the token's `id_token` claims (or userinfo response) and reject if false.
- **`httpx` sync vs async**: the Tiangolo template uses sync routes with a sync DB session; keeping `httpx.Client` (sync) avoids introducing `asyncio` complexity in route handlers.

## Migration Plan

1. Add Alembic migration: `ALTER TABLE "user" ALTER COLUMN hashed_password DROP NOT NULL`.
2. Add guard in `POST /login/access-token`: reject users with `hashed_password IS NULL`.
3. Deploy backend with new endpoints and migration applied.
4. No rollback complexity: making the column nullable again is reversible as long as no `NULL` rows were inserted (rollback: re-run migration with `NOT NULL`).

## Open Questions

- Should the state cookie be `Secure` and `SameSite=Lax`? Yes — requires HTTPS in staging/production; local dev can relax this.
- Should we scope the Google OAuth to specific hosted domains (`hd` param)? Not for v1 (any Google account is accepted).
